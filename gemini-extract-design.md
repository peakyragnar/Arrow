System Architecture and Data Taxonomy for Automated SEC EDGAR XBRL Extraction1. Executive Introduction to EDGAR Modernization and Programmatic Data ExtractionThe United States Securities and Exchange Commission (SEC) Electronic Data Gathering, Analysis, and Retrieval (EDGAR) system operates as the central nervous system for corporate transparency, housing millions of financial statements, material event reports, and registration documents. Historically, the extraction of fundamental financial data from EDGAR required sophisticated, heuristic-driven natural language processing and brittle web-scraping techniques designed to parse unstructured ASCII or HyperText Markup Language (HTML) documents. These methodologies were inherently fragile, as they relied heavily on matching volatile text strings, navigating inconsistent table structures, and guessing the contextual meaning of specific line items based on their spatial proximity to other data points.The mandate for Extensible Business Reporting Language (XBRL) and its successor, Inline XBRL (iXBRL), revolutionized this landscape by embedding machine-readable, heavily structured metadata directly within the human-readable formatting of financial disclosures. By associating every individual financial fact—ranging from a top-line revenue figure to a granular, text-based accounting policy—with a standardized, globally recognized digital definition, iXBRL eliminates the ambiguity of corporate reporting. However, to construct an automated system capable of performing blanket downloads of any SEC filing and subsequently instructing a large language model (LLM) or open-source artificial intelligence (AI) to extract the Income Statement, Balance Sheet, and Cash Flow Statement in strict accordance with defined accounting standards, a deep architectural integration is required.This comprehensive technical report serves as the definitive blueprint for building such a system. It details the precise rules, formats, infrastructural topologies, and taxonomies required to engineer a robust ingestion pipeline and prompt architecture. By mastering the EDGAR network protocols, traversing the complex Standard Generalized Markup Language (SGML) container files, parsing the semantic graphs of XBRL linkbases, and enforcing the strict definitional logic of the US GAAP Financial Reporting Taxonomy (GRT), developers can provide AI models with the precise training data and operational constraints necessary to yield pure, mathematically validated, and globally comparable financial outputs, regardless of idiosyncratic corporate formatting.2. EDGAR Network Infrastructure and Programmatic Ingestion PipelinesTo enable a system to blanket-download SEC filings, developers must navigate a complex ecosystem of real-time Application Programming Interfaces (APIs), bulk archive protocols, legacy file directory hierarchies, and strict rate-limiting governance. Constructing an ingestion pipeline requires a nuanced understanding of when to employ real-time scraping versus bulk archival downloading.2.1 Fair Access Policies, Rate Limiting, and Throttling ProtocolsThe SEC strictly enforces automated access policies to protect the integrity and availability of the EDGAR infrastructure. Programmatic access is governed by a hard rate limit of no more than 10 requests per second across all SEC domains originating from a single user or entity, regardless of the number of machines or IP addresses used to submit the requests. The maximum allowable egress rate is generally capped at approximately 30 megabytes per second.Any automated system must implement sophisticated throttling mechanisms to respect these limits. If a crawler exceeds the 10-request-per-second threshold, the SEC's perimeter security will classify the traffic as a botnet or abusive automated tool, resulting in immediate connection resets, HTTP 403 Forbidden errors, and temporary or permanent IP bans. Furthermore, every HTTP request must include a compliant User-Agent header. The SEC mandates that this header statically declare the application name and a valid contact email address (e.g., User-Agent: 'Mozilla/5.0... SEC-Downloader your.email@example.com'). Failure to supply a correctly formatted User-Agent string will result in immediate request rejection. The domains do not support Cross-Origin Resource Sharing (CORS), requiring all API interactions to be handled server-side rather than directly from browser-based client applications.2.2 RESTful JSON APIs versus Bulk Data ArchivesThe SEC has modernized data access by providing RESTful data APIs hosted on data.sec.gov, delivering pre-formatted JavaScript Object Notation (JSON) payloads. These APIs aggregate XBRL facts and submission histories without requiring API key authentication. For a system designed to extract financial statements for arbitrary companies, these endpoints offer a highly structured, though rate-limited, retrieval mechanism.Table 1 delineates the primary RESTful endpoints available for structured data extraction.API Endpoint PathOutput DescriptionUpdate FrequencySource Reference/submissions/CIK##########.jsonProvides the filing history for a specific entity. For system initialization and foundational LLM training data generation, the SEC provides nightly bulk archive ZIP files. The companyfacts.zip archive contains the complete output of the XBRL Frame and Company Facts APIs for all companies, while the submissions.zip archive provides the exhaustive public EDGAR filing history for all registered filers. Ingesting these archives allows developers to populate a localized, high-performance database, completely circumventing SEC rate limits for historical data.2.3 RSS Feeds, Ingestion Logs, and Index ArchivesTo maintain synchronization with the SEC after the initial bulk download, the system must parse Really Simple Syndication (RSS) feeds and daily index files. The SEC publishes RSS feeds for EDGAR structured disclosure submissions, updated every ten minutes from 6:00 AM to 10:00 PM Eastern Standard Time. These feeds provide real-time metadata (company name, accession number, CIK, form type) and direct links to the compressed ZIP files containing the interactive iXBRL data.Alternatively, the EDGAR Index APIs provide access to daily ingestion logs and file index archives. The File Index Archive API returns gzip-compressed JSON Lines (.jsonl.gz) files. Each line in these files is a self-contained JSON object representing a single EDGAR filing and all associated published files, allowing an automated crawler to precisely track every R*.htm or .xml document uploaded on a given day without needing to brute-force directory listings.2.4 Navigating the EDGAR File System Hierarchy and SGML ContainersWhen a system attempts to download the raw filing documents directly, it must interface with the EDGAR Archives directory structure. Entities are identified by a Central Index Key (CIK), a unique numerical identifier that is never recycled. Individual filings are identified by an Accession Number, a 20-character string formatted as CIK-Year-Sequence (e.g., 0001193125-15-118890), which is automatically assigned upon SEC acceptance.The raw text version of the complete disseminated filing is located via an alternative symbolic path that strips the dashes from the accession number to create an intermediate directory: /Archives/edgar/data/{CIK}/{AccessionNumberWithoutDashes}/{AccessionNumber}.txt.It is highly critical for developers to understand that this .txt file is not a standard text document. It is a legacy Standard Generalized Markup Language (SGML) container bundle. SGML, an international standard (ISO 8879) that predates and gave rise to HTML and XML, is used by the SEC to bundle dozens of disparate files—including primary HTML filings, XBRL XML schemas, graphic images, and index files—into a single, transmittable package.The SEC employs two distinct SGML container formats. The non-public complete file (.nc) acts as an internal submission manifest used by the SEC for initial validation. The public-facing complete submission text file (.txt) contains the entire content payload. The SGML structure utilizes tags such as <SEC-DOCUMENT>, <SEC-HEADER>, and <ACCEPTANCE-DATETIME>, followed by repetitive <DOCUMENT> nodes that encapsulate the individual base64-encoded or raw text files.To prepare data for AI extraction, the ingestion pipeline must deploy an SGML parser to separate this monolith into its constituent files. Regular expression extraction (e.g., ^<(.+?)>(.+?)$ with multiline options) can be utilized to quickly map the SGML headers into a dictionary payload, separating the primary financial statements from the noise of exhibits and graphics.3. Anatomy of an SEC Submission: Document Types and XBRL SpecificationsOnce the SGML bundle is downloaded and unpacked, the AI pipeline must navigate a chaotic folder of randomly named files. Before feeding data into a localized LLM, the system must programmatically isolate the specific documents that contain the financial statements, while ensuring the files adhere to strict EDGAR validation rules.3.1 Navigating via the FilingSummary.xml Table of ContentsA typical 10-K or 10-Q submission contains dozens of files, and companies employ wildly inconsistent naming conventions for their primary HTML and XML documents. Attempting to locate the Balance Sheet or Income Statement by searching file names or HTML titles is highly error-prone.To solve this, the pipeline must exclusively rely on the FilingSummary.xml file, which is automatically generated by EDGAR processing software and included in post-2011 submissions. This XML document acts as the master table of contents for the interactive data. It maps human-readable report names (e.g., "Consolidated Statements of Operations") to their specific physical files (e.g., R2.htm or R3.xml). The AI training logic should instruct the ingestion engine to parse the FilingSummary.xml using fuzzy matching on standard accounting titles, identify the corresponding R*.htm references, and only forward those specific files into the extraction prompt window.3.2 HTML and ASCII Technical ConstraintsThe files extracted from the SGML bundle are subject to rigid, antiquated formatting constraints. While the broader internet operates on HTML5, the SEC mandates that all filings strictly conform to the HTML 3.2 standard, with severe restrictions on embedded styling and active content. Extraneous tags are prohibited, and nested tables are heavily constrained, meaning the visual presentation of financial statements is often achieved through archaic spacing and fixed-width alignment.Furthermore, files submitted as ASCII text are restricted to a highly limited subset of valid characters. Characters outside this set, including advanced word-processing smart quotes, em-dashes, or non-standard control codes, are invalid and stripped from the document. For the AI, this means that parsing visual HTML tables is a computationally expensive and unreliable method for data extraction. The visual layout cannot be trusted; the system must pivot entirely to extracting the underlying iXBRL tags.3.3 Core XBRL Specifications and RegistriesThe EDGAR XBRL environment is governed by specific versions of XBRL International (XII) standards. Filers are not permitted to use arbitrary XML; they must adhere to the EDGAR Filer Manual (EFM) Volume II Chapter 6. The technical syntax must comply with XBRL version 2.1, XBRL Dimensions 1.1, XBRL Calculations 1.1, and Inline XBRL 1.1. Additionally, the SEC supports the XBRL Open Information Model (OIM) 1.0, which allows for xBRL-XML mapping, enabling developers to translate convoluted XML structures into highly streamlined, JSON-formatted data arrays prior to AI processing. The system must also account for specific XII registries, including the Data Types Registry (DTR) and Unit Types Registry (UTR), which dictate how dates, currencies, and share counts are encoded.4. Dissecting the iXBRL Markup Syntax for AI ParsingInline XBRL (iXBRL) allows a single document to serve both as a human-readable HTML file and a machine-readable data repository. It achieves this by wrapping financial figures and text strings in specific <ix:...> namespace tags. To train an open-source model to extract information correctly, the AI must understand the behavior, attributes, and transformational rules of these tags.4.4 Numeric Facts and the ix:nonFraction TagWhen a company reports a numeric value, such as a revenue figure of $500 million, the number visible on the webpage might simply be "500". The iXBRL tag ix:nonFraction encapsulates this number and provides the critical metadata required to determine its true mathematical value.Table 2 details the mandatory attributes of the ix:nonFraction tag that the AI extraction logic must process.iXBRL AttributeTechnical Purpose and AI Processing LogicSource ReferencenameIdentifies the exact taxonomy element being reported (e.g., us-gaap:Revenues). The AI uses this as the primary key for categorization.contextRefLinks the fact to a specific context definition (defined elsewhere in the document), detailing the reporting period (duration vs. instant) and the entity identifier.unitRefLinks the fact to a defined unit of measure (e.g., ISO 4217 standard currency codes like iso4217:USD, or shares).decimalsIndicates the precision of the reported number. The EDGAR system restricts the use of this attribute; it cannot be used to truncate non-zero digits.scaleA critical transformational attribute expressing a power of ten. The AI must mathematically multiply the extracted inner text by $10^{scale}$. If the text is "500" and the scale is "6", the AI must output $500,000,000$.signIf the value is visually presented as a positive number but implies a negative accounting impact (e.g., expenses shown without parentheses), a sign="-" attribute reverses the polarity.An AI model cannot simply scrape the visible text; it must execute the formula: $Output = (Extracted\_Text \times 10^{scale}) \times (Sign\_Multiplier)$.4.5 Textual Facts, ix:nonNumeric, and the escape AttributeFinancial statements are not purely numerical; they include extensive footnotes, accounting policy text blocks, and descriptive strings. These are tagged using ix:nonNumeric. The most critical attribute for an AI processing text blocks is the escape attribute, which dictates how embedded HTML should be handled.If escape="true", the Inline XBRL specification mandates that any HTML tags occurring within the text block must be included in the resulting fact value, yielding a valid XHTML fragment. This is typically used for textBlockItemType facts, preserving the visual formatting of complex tables within footnotes. If the AI is tasked with extracting an entire accounting policy for semantic analysis, it must know to retain this XHTML structure.Conversely, if escape="false", the tag is asserting that the content is plain text, typically associated with stringItemType datatypes. In this scenario, all embedded HTML tags are stripped. However, the AI must apply complex whitespace normalization. Web browsers collapse multiple spaces, tabs, and carriage returns into a single space. Therefore, <span>My</span><span>policy</span> might render as "My policy" in a browser, but the raw extracted value with escape="false" would be "Mypolicy" without spaces. The extraction algorithm must be meticulously programmed to preserve word boundaries when converting iXBRL to plain text to prevent corrupting the LLM training data.4.6 Continuation Tags and Fact ConcatenationCompanies frequently encounter situations where a single accounting policy or footnote spans multiple pages or is interrupted by page breaks and tables. To maintain the integrity of a single XBRL fact, the preparer splits the text across multiple ix:nonNumeric tags and links them using the continuedAt and id attributes.The first tag will contain continuedAt="PolicyPart2". The subsequent tag on the next page will bear id="PolicyPart2". The AI ingestion pipeline must contain recursive logic that hunts for these attributes, concatenating the isolated text fragments in memory before presenting the finalized, uninterrupted string to the language model for analysis.5. The US GAAP Financial Reporting Taxonomy (GRT) GuidelinesTo instruct the open-source model to extract financial statements "correctly as per the definitions," the system cannot rely on the company's visual presentation. The AI must be hardcoded with the structural rules of the Financial Accounting Standards Board (FASB) US GAAP Financial Reporting Taxonomy (GRT), which consists of over ten thousand heavily documented elements.The prompt architecture must instruct the AI to hunt for specific, mandatory us-gaap nodes to construct the three primary financial statements.6.1 The Balance Sheet (Statement of Financial Position)The Balance Sheet represents the entity's financial position at an instant in time. Elements here possess a natural balance type: Assets are debits, while Liabilities and Equity are credits.The AI must be trained to extract the following hierarchical structure, traversing down the calculation linkbase to find the constituent children:us-gaap:AssetsCurrent (Debits: e.g., Cash and Cash Equivalents, Accounts Receivable, Inventory) us-gaap:AssetsNoncurrent (Debits: e.g., Property, Plant, and Equipment) us-gaap:Assets (The master debit total)us-gaap:LiabilitiesCurrent (Credits: e.g., Accounts Payable, Short-term Debt) us-gaap:LiabilitiesNoncurrent (Credits: e.g., Long-term Debt, Deferred Tax Liabilities) us-gaap:Liabilities (The master liability total)us-gaap:StockholdersEquity (or us-gaap:PartnersCapital for partnerships, a credit total) us-gaap:LiabilitiesAndStockholdersEquityThe AI extraction engine must autonomously enforce the fundamental accounting equation:$$us-gaap:Assets = us-gaap:Liabilities + us-gaap:StockholdersEquity$$
If the data extracted from the instance document violates this mathematical rule, the AI should trigger a self-correction loop, re-evaluating the Calculation Linkbase to locate misplaced custom extensions or signs flipped by incorrect ix:nonFraction processing.6.2 The Income Statement (Statement of Operations)The Income Statement measures financial performance over a defined duration (e.g., three months, twelve months). The taxonomy structure cascades down to a definitive bottom line.The AI prompt should target this specific element cascade:us-gaap:Revenues or us-gaap:SalesRevenueNet (The top-line credit)us-gaap:CostOfGoodsAndServicesSold (Debit)us-gaap:GrossProfitus-gaap:OperatingExpenses (Debit: e.g., Selling, General and Administrative, Research and Development) us-gaap:OperatingIncomeLossus-gaap:IncomeLossFromContinuingOperationsBeforeIncomeTaxesMinorityInterestAndIncomeLossFromEquityMethodInvestments (A critical intermediate node frequently subject to Data Quality Committee checks like DQC 0119) us-gaap:NetIncomeLoss (The final bottom-line result) 6.3 The Statement of Cash FlowsThe Cash Flow Statement reconciles net income to the actual cash generated or utilized during the reporting duration, categorized strictly into operating, investing, and financing activities. Constructing training data for the Cash Flow Statement requires the AI to navigate severe taxonomy anomalies and international reporting discrepancies.The standard targeted tags include:us-gaap:NetCashProvidedByUsedInOperatingActivitiesus-gaap:NetCashProvidedByUsedInInvestingActivitiesus-gaap:NetCashProvidedByUsedInFinancingActivitiesus-gaap:CashAndCashEquivalentsPeriodIncreaseDecrease 6.3.1 The OperatingActivities Balance Type AnomalyThe AI must be explicitly programmed to handle a known structural limitation within the XBRL Calculation Linkbase regarding cash flows. Under strict accounting rules, cash inflows are debits and outflows are credits. However, in the US GAAP taxonomy, the element us-gaap:NetCashProvidedByUsedInOperatingActivities is intentionally modeled without any assigned balance type (neither debit nor credit).This anomaly exists because operating cash flow is predominantly presented using the indirect method. This method begins with Net Income (a credit balance element) and reconciles it by adding back non-cash expenses such as depreciation (a debit balance element). The XBRL calculation engine strictly prohibits the addition of a debit element to a credit element. To circumvent this mathematical limitation and allow the calculation weights to align with reality, the FASB stripped the balance type from the operating cash flow element. If an AI attempts to enforce standard debit/credit validation on this specific subsection of the taxonomy, it will generate thousands of false-positive errors.Furthermore, the system must integrate XBRL US Data Quality Committee (DQC) rules into the AI's validation constraints. Specifically, DQC 0044 ensures that accrual-based elements—which are designed exclusively for the Income Statement or Statement of Stockholders' Equity—are not inappropriately mapped into the Cash Flow Statement reconciliation.6.3.2 Cross-Border Differences: US GAAP versus IFRSIf the AI system is tasked with parsing foreign private issuers filing under International Financial Reporting Standards (IFRS) via forms like 20-F or 40-F , it must swap its definitional logic. While US GAAP mandates net income as the starting point for indirect cash flows, IFRS (IAS 7) allows for variations, including starting from profit before tax or operating profit. Additionally, under US GAAP, interest paid and received must be classified as operating cash flows, whereas IFRS permits entities to classify interest paid as financing and interest received as investing. The prompt architecture must read the root taxonomy namespace (us-gaap: versus ifrs-full:) and dynamically load the corresponding accounting definitions into the context window.7. Algorithmic Resolution of Custom Taxonomies (Extensions and Anchoring)The most formidable barrier to extracting standardized financial data across thousands of companies is the pervasive use of custom extension taxonomies. While the SEC strongly encourages the use of standard elements, corporate preparers are permitted—and frequently required—to create custom tags when standard elements do not accurately capture the nuance of a highly specific transaction or business segment.If an AI is rigidly prompted to search only for us-gaap:Revenue, it will completely fail to extract top-line data from a technology company that has created a custom tag such as custom:CloudSubscriptionAndSupportRevenue. To ensure the AI outputs financial statements "correctly as per the definitions," the system must not discard custom data; it must algorithmically resolve these extensions back into the standardized US GAAP hierarchy. This is achieved by programming the AI to traverse two distinct linkbase pathways: Calculation Roll-ups and ESEF Definition Anchoring.7.1 Resolution via ESEF Wider-Narrower Anchoring (Definition Linkbase)Modern XBRL taxonomies have adopted "anchoring" mechanisms, heavily influenced by the European Single Electronic Format (ESEF) reporting rules, which mandate that all custom extension concepts be explicitly linked to the standard base taxonomy concept possessing the "closest wider accounting meaning and/or scope". This creates a machine-readable breadcrumb trail leading from the bespoke corporate tag back to the universal standard.The AI must parse the Definition Linkbase for these wider-narrower arcs to resolve anomalies. This linkbase establishes strict summation-item arcs representing parent-child mathematical relationships.If the AI encounters an unrecognized custom:SpecializedInventory tag, it queries the calculation graph to locate the element's parent. If the graph indicates that custom:SpecializedInventory has a weight of +1.0 and rolls up into the parent element us-gaap:InventoryNet, the AI logically deduces that the custom fact is a disaggregated component of inventory. During the generation of the standardized Balance Sheet, the AI seamlessly aggregates this custom value under the standard us-gaap:InventoryNet node, completely neutralizing the company's non-standard reporting format.8.
