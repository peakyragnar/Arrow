# ADR-0003: uv for Python package and environment management
Status: accepted
Date: 2026-04-19

## Context

Arrow is a greenfield Python project as of step 2. We needed a toolchain for:
- Dependency declaration (what packages, what versions)
- Lockfile discipline (reproducible installs across machines / CI / production)
- Virtual environment management
- Python version pinning

Three realistic options:

- **pip + `requirements.txt`** — minimal; no lockfile, no resolver, no venv management
- **Poetry** — `pyproject.toml` + lockfile + resolver + venv + publishing; was the default 2020–2023
- **uv** — Astral's tool (same team as ruff); `pyproject.toml` (PEP 621) + `uv.lock` + venv + Python version management

## Decision

**uv.** Project scaffolded with `uv init`. `pyproject.toml` declares dependencies per PEP 621; `uv.lock` pins exact versions; `.python-version` pins the interpreter (currently 3.14). Dev dependencies declared under `[dependency-groups] dev`.

Runtime commands:
- `uv sync` — install from lockfile
- `uv run python scripts/foo.py` — execute inside the managed venv without manual activation
- `uv add <pkg>` — add and resolve dependencies
- `uv add --dev <pkg>` — add a dev-only dependency

## Consequences

**Positive**
- Fast resolver (~10–100× Poetry's) — noticeable on every dependency change
- Standard PEP 621 `pyproject.toml` — not a Poetry-specific dialect, portable if we ever switch tools
- Python version managed by the same tool (`.python-version`) — no separate pyenv install
- Lockfile discipline from day one; reproducible across fresh clones

**Negative**
- Tool is relatively new (released 2024) — corner cases may still surface; less institutional Stack Overflow tail than pip or Poetry
- Ecosystem coherence assumes Astral remains invested; organizational risk exists though it hasn't materialized

## Alternatives considered

- **pip + `requirements.txt`** — fine for scripts and glue code, not for a project expected to live 12+ months with growing deps. No lockfile means environment drift is inevitable.
- **Poetry** — mature, but slow resolver, opinionated defaults, and its pyproject.toml dialect is not PEP 621. The Python ecosystem has largely moved on from Poetry as the default for new projects.

## When to revisit

- Astral abandons uv or the tool stagnates for 12+ months
- A corner-case blocker surfaces that's materially cheaper to fix by switching than by waiting
- An organizational constraint appears that forbids tools younger than 2 years

Migration to pip or Poetry: `pyproject.toml` mostly carries over (PEP 621 is standard). The `uv.lock` is uv-specific and would be regenerated by the new tool. Low cost.
