# ship-schema-change

Ship an Arrow schema migration atomically with its docs. Enforces the rule from
[AGENTS.md](../../AGENTS.md):

> Schema changes ship with their docs. A migration that adds, removes, or
> supersedes a table updates the v1 Tables status table in
> `docs/architecture/system.md` and any reference-doc mentions in the same
> commit. ADRs about withdrawn or superseded tables get their status flipped,
> never their bodies edited. Build Order step markers get updated too. After
> applying a migration, re-run `uv run scripts/gen_schema_viz.py` to regenerate
> the live schema view at `arrow_db_schema.html`.

This skill turns that rule into a checked workflow. Doc drift is the failure
mode; verification checkpoints are the guardrail.

---

## When to use

Trigger this skill whenever you are about to:

- Create a new file in [db/schema/](../../db/schema/)
- Modify an existing migration file
- Rename, drop, or supersede a table or column
- Retroactively clean up a schema change that shipped without docs
- Add a column to an existing table

If unsure whether a change qualifies: if it touches DDL, it qualifies.

---

## Step 1 — Classify the change

Classification decides which docs get touched downstream. Pick one:

| Category | Example | Docs touched |
|---|---|---|
| **new-table** | add `prices_daily` | system.md v1 Tables row + Build Order marker + any reference doc that names it |
| **new-column** | add `currency` to `financial_facts` | system.md Table Intent block (if documented there) + reference docs that name the column |
| **rename** | `foo` → `bar` | same as new + sweep all docs/ and src/ for old name |
| **supersede** | `artifact_chunks` → `artifact_section_chunks` | status flip on old table row + ADR status flip (body untouched) + dependent-flag cleanup per supersession rule |
| **drop** | remove `artifact_chunks` | flip status to `withdrawn` with migration reference + related ADR status flip if applicable |

Compound changes (e.g. "supersede table X by introducing table Y") are
classified as **supersede** and do both the add and the flip.

---

## Step 2 — Write the migration

- Find the highest number in [db/schema/](../../db/schema/):
  `ls db/schema/ | tail -5`
- Next file: `NNN_short_description.sql` where `NNN` is the next zero-padded
  number.
- Follow the style of recent migrations
  ([014_sec_qualitative_layer.sql](../../db/schema/014_sec_qualitative_layer.sql),
  [015_artifact_text_units.sql](../../db/schema/015_artifact_text_units.sql))
  for statement ordering, comments, and naming.
- Per
  [ADR-0004](../../docs/decisions/0004-hand-rolled-migrations.md):
  migrations are hand-rolled — no ORM, no autogenerated SQL.
- Per
  [ADR-0009](../../docs/decisions/0009-supersedes-restrict-not-set-null.md):
  FK supersession uses `ON DELETE RESTRICT`, not `SET NULL`.

---

## Step 3 — Apply the migration

```bash
uv run scripts/apply_schema.py
```

Verify the output names the new migration file. A silent "No pending
migrations" means your file was not discovered — check the filename.

---

## Step 4 — Update the docs

Always touch these, driven by classification:

### Every change
- **[docs/architecture/system.md](../../docs/architecture/system.md)** — v1
  Tables status table (around the section titled `## v1 Tables`). For new
  tables: add a row. For supersessions: flip the old row's status to
  `withdrawn` or `superseded` with a migration pointer. For renames: update
  the row name and annotate.
- **Build Order** section in the same file. If this change completes, advances,
  or retires a numbered step, update its marker (✅ 🚧 ⏳ ⬜) and append the
  migration number.

### Reference docs — grep sweep
Run the sweep and visit each hit:
```bash
grep -rn "<table_or_column_name>" docs/reference/ docs/architecture/
```
Common reference doc targets:
- [docs/reference/concepts.md](../../docs/reference/concepts.md) — canonical
  bucket contract
- [docs/reference/fmp_mapping.md](../../docs/reference/fmp_mapping.md) — FMP ↔
  canonical mapping
- [docs/reference/artifact_metadata.md](../../docs/reference/artifact_metadata.md)
  — artifact metadata keys
- [docs/reference/formulas.md](../../docs/reference/formulas.md) — metric
  formulas (update if a dependent column changed)

### ADRs (supersede / drop only)
- Find the ADR that discusses the removed/superseded table in
  [docs/decisions/](../../docs/decisions/).
- Flip the `- Status:` line in the body to `superseded` or `withdrawn`.
- **Do not edit the ADR body.** Context is frozen; only the status moves.
- If the ADR references the migration that retired it, you may append a
  single line under the status marker pointing at the new migration.

### Supersession-specific
Per the supersession-cleanup rule: re-ingest or migrations that supersede
facts must resolve dependent flags in the same pass. If your change
supersedes rows, ensure dependent `data_quality_flags` or references are
resolved — do not leave dangling pointers.

---

## Step 5 — Regenerate the schema viz

```bash
uv run scripts/gen_schema_viz.py
```

Confirm [arrow_db_schema.html](../../arrow_db_schema.html) actually changed:
```bash
git diff --stat arrow_db_schema.html
```
A zero-byte diff means either the migration did not apply or the viz
script is reading a stale database. Investigate before moving on.

---

## Step 6 — Run verification checkpoints

These are the "flags" for this skill. Each must pass before commit. If any
fails, return to the step implied by that failure.

| # | Check | Command | Pass criteria |
|---|---|---|---|
| 1 | Migration applied | `psql -c "SELECT name FROM schema_migrations ORDER BY name DESC LIMIT 3"` (via `uv run scripts/db_ping.py` or direct psql per setup.md) | New migration filename appears |
| 2 | Table exists (add/rename) | `psql -c "SELECT to_regclass('<table_name>')"` | Returns the table name, not NULL |
| 3 | Table removed (drop) | same query | Returns NULL |
| 4 | system.md v1 Tables updated | `grep -c "<table_name>" docs/architecture/system.md` | ≥ 1 (add) or 0 except in historical note (drop) |
| 5 | Reference doc sweep clean | `grep -rn "<old_name>" docs/` (for rename/drop) | Zero matches, or only in changelog/ADR context |
| 6 | ADR status flipped (supersede/drop) | `grep -n "Status:" docs/decisions/<relevant-adr>.md` | Shows `superseded` or `withdrawn` |
| 7 | Schema viz regenerated | `git diff --stat arrow_db_schema.html` | Non-zero line delta |
| 8 | No dangling references | `grep -rn "<old_name>" src/ db/ tests/` (for rename) | Zero matches except migration files that legitimately reference both old and new |

Skip rows that don't apply to the classification.

---

## Step 7 — Commit atomically

Per Arrow workflow: commit directly to main, one commit for the whole
schema change. Stage only what this change touched:

```bash
git status
# Stage the migration, the doc updates, and the regenerated schema viz
git add db/schema/NNN_*.sql \
        docs/architecture/system.md \
        arrow_db_schema.html \
        <any reference docs touched> \
        <any ADR touched>
git diff --cached --stat   # sanity-check what's in the commit
git commit -m "Add/supersede/drop <table>: <brief rationale>"
```

Commit message shape:
- First line: what changed and which table/column
- Body (if any): why — business or architecture reason, not restating the
  diff

Do **not** amend the preceding commit, and do **not** use worktrees or
feature branches (per project workflow).

---

## Failure modes this skill is designed to catch

- Migration applied but docs never updated → caught by checkpoints 4-6
- ADR body edited instead of status flipped → caught by careful reading of
  Step 4 (watch for accidental body edits in `git diff`)
- Schema viz forgotten → caught by checkpoint 7
- Stale mention of the old table name in a reference doc after a rename →
  caught by checkpoint 5
- Supersession leaves dangling dependent flags → caught by the explicit
  supersession-cleanup call-out in Step 4
- Migration file created but `apply_schema.py` did not pick it up (usually
  wrong filename prefix) → caught by checkpoint 1

If a checkpoint fails twice in a row on the same step, stop and read the
relevant source file directly. Do not keep re-running blindly.

---

## Out of scope

- Populating the new table with data (that's an ingest skill, to be written
  separately).
- Changing ADR decisions themselves (open a new ADR; do not edit an
  accepted one).
- View migrations (`db/queries/*.sql`) unless they are driven by a table
  change — those may warrant a separate companion skill.
